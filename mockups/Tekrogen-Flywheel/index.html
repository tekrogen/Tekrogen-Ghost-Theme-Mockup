07 · ebia · personal finance

Token-safe Plaid integration on a four-week budget.

Why we kept access tokens server-side, what we gave up to do it, and the revocation path that actually shipped — for the engineer who has to maintain it.

next.jsserver actionsplaidpostgres

The Problem

01

EBIA needed to link real bank accounts on day one, but the team was four engineers on a four-week budget. The obvious path — a hosted aggregator widget that stored tokens for us — failed the one constraint we couldn't move: a breach of our database must not expose anyone's banking credentials. The question wasn't "which vendor," it was "where does the secret live, and who is liable when it leaks."

Acceptance Criteria

02
  • No long-lived access token ever reaches the browser.
  • A database dump alone cannot reconstruct a usable token.
  • Revocation propagates within one webhook cycle.
  • The whole exchange fits in a Server Action — no extra service.

The Stack Decision

03

We scouted four approaches in the recon note (linked in the sidebar) and chose server-only session handling: the browser holds a short-lived public link token, the access token is exchanged and used entirely inside a Server Action, and only an encrypted session handle is persisted. The decision belonged to the constraints — a small team that can defend a small surface — not to a feature score.

Key Trade-offs

04
We choseWe gave up
Server-only token exchangeRe-auth when a session is lost; no offline token reuse.
Encrypted session handleAn extra KMS dependency in the deploy.
Webhook-driven revocationEventual consistency — a few seconds of lag.

The Build

05

The exchange runs in a single Server Action; the public link token is the only thing the client ever holds. Revocation arrives by webhook, invalidates the encrypted handle, and is queued with retry + backoff so a transient Plaid outage can't drop a revocation on the floor. Multi-item households aggregate several linked accounts per user behind one handle.

What We'd Do Differently

06

We'd move the queue off the app process sooner. It was fine at pilot scale, but the moment a second product wanted the same revocation guarantees, the in-process queue became the thing we had to extract. The hardened build ships that separation by default.

The Outcome

07

Shipped in the four-week window. Zero tokens at rest, revocation inside one webhook cycle, and a clean enough boundary that the whole thing became the first paid template on .com. The replay of the reasoning is below.

Four wings, one pursuit.

Build it, prove it, write down what we learned. Tekrogen documents real implementations as transparent case studies — then packages the result as demos and production templates. Not designed to sell; published because it works.

how we work

Principles

01

Honest trade-offs

Every study documents what we gave up, including what failed. No retrospective is sanitized for marketing.

02

Real use cases only

Every solution originates from a build we actually used in practice.

03

No scores

Recon is qualitative. The final stack call belongs to the constraints, not a number.

04

Data minimization

We collect only what's needed to deliver a license. No behavioral data sold or shared.
